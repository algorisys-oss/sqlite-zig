# What is *not* migrated (and why)

The port covers the **entire active Linux engine** — 90 modules in `src/*.zig`
(pager, B-tree, VDBE, the full SQL compiler, OS/VFS, and every enabled extension:
R-Tree/Geopoly, Session, FTS3/4/5, JSON). Of the **102 active C translation
units** in [`../vendor/tu.txt`](../vendor/tu.txt), **12 remain as C**.

None of the 12 are part of the hand-written engine logic. They fall into three
buckets. To regenerate this list:

```bash
ported=$(grep -oE '"[a-z0-9_]+\.c"' build.zig | tr -d '"' | sort -u)
comm -23 <(grep -oE '[a-z0-9_]+\.c' vendor/tu.txt | sort -u) <(echo "$ported")
```

## 1. Generated — actively used, but ported *from the generator*, not by hand (2)

These two execute in a normal Linux build, but are machine-generated tables, not
source we author. [../CLAUDE.md](../CLAUDE.md) is explicit: port from the
generator/grammar, **never** hand-port a generated file.

| File | What it is | Reason |
|---|---|---|
| `parse.c` | The SQL parser state machine (~6k lines of tables) | Generated by **Lemon** from `parse.y`. Hand-porting it would mean maintaining a transliterated copy that drifts from the grammar; the right move is to port the Lemon-generated *output* mechanically or run Lemon. The tokenizer that feeds it (`tokenize.c`) **is** already Zig (`src/tokenize.zig`), and calls the generated `sqlite3Parser`. |
| `opcodes.c` | VDBE opcode names + the per-opcode property byte array | Generated from `vdbe.c` by `tool/mkopcodeh.tcl`. Pure data tables keyed to the opcode numbers `vdbe.zig` already uses. |

➡️ **The only remaining C that runs in the default build.** A "100% pure-Zig
Linux engine" is blocked solely on these two generated tables, not on logic.

## 2. Windows-only — never compiled into a Linux engine (2)

Excluded by the OS-selection macros (`SQLITE_OS_WIN == 0` on Linux); the bodies
compile away. Porting them is only relevant if we target Windows.

| File | What it is |
|---|---|
| `os_win.c` | The Windows VFS (file I/O, locking, mmap) |
| `mutex_w32.c` | Windows critical-section mutexes |

The Linux equivalents **are** ported: `src/os_unix.zig`, `src/mutex_unix.zig`.

## 3. Flag-inactive alternates / extensions — compile to empty stubs (8)

Each is guarded by a feature macro that **this build does not define** (see
`sqlite_flags` in [../build.zig](../build.zig)), so the file compiles to a
near-empty object and contributes no runtime code. Enabling the feature is what
would require porting it.

| File | Gated on (not enabled here) | Notes |
|---|---|---|
| `mem0.c` | `SQLITE_ZERO_MALLOC` | A no-op allocator for builds with no heap. |
| `mem2.c` | `SQLITE_MEMDEBUG` | Debug allocator with guard bytes. |
| `mem3.c` | `SQLITE_ENABLE_MEMSYS3` | Alternate allocator — we enable **memsys5** instead, and `mem5.c` **is** ported. |
| `notify.c` | `SQLITE_ENABLE_UNLOCK_NOTIFY` | `sqlite3_unlock_notify` API. |
| `os_kv.c` | (kvvfs) | The key/value VFS used by the WASM/browser build, not the unix VFS. |
| `icu.c` | `SQLITE_ENABLE_ICU` | ICU collations/functions. |
| `fts3_icu.c` | `SQLITE_ENABLE_ICU` | ICU tokenizer for FTS3. |
| `sqlite3rbu.c` | `SQLITE_ENABLE_RBU` | Resumable Bulk Update. |

## Bottom line

| Bucket | Count | Runs in the default Linux build? | Blocks "pure Zig"? |
|---|---:|---|---|
| Generated tables (`parse.c`, `opcodes.c`) | 2 | **Yes** | Only these — and they're generated, not logic |
| Windows VFS/mutex | 2 | No (Linux uses the ported unix code) | No, unless targeting Windows |
| Flag-inactive alternates/extensions | 8 | No (stubs under our flags) | No, unless enabling that feature |
| **Total not migrated** | **12** | | |

Everything that constitutes the running engine on Linux — all hand-written
logic — is Zig. See [../PROGRESS.md](../PROGRESS.md) for the ported-module list
and [testing.md](testing.md) for how correctness is verified.
